学习笔记之'use strict'

发表于 2016-11-20 |

概念

如果给javascript代码标志为“严格模式”，则其中运行的所有代码都必然是严格模式下的。
其一：如果在语法检测时发现语法问题，则整个代码块实效，并导致一个语法异常。
其二：如果在运行期出现了违反严格模式的代码，则抛出执行异常。
注：经过测试ie6,7,8,9均不支持严格模式。
使用方法，添加“use strict”
在如下位置加入可以开启相应代码块中的严格模式：
1、必须在全局代码的开始处加入。
2、在eval代码开始处加入。
3、在函数声明代码开始处加入。
4、在 new function（）所传的body参数块开始加入。
例1:

在非严格模式下可以使用0开头的前缀声明8进制。显示10。
但在严格模式下，会产生错误。

如果是用严格模式，除了0开头前缀8进制以外的还有：
1、在代码中不能使用一些扩展的保留子：
implements,interface,let,package,private,public,static,yield
2、with语句也不能使用。
3、不能声明或重写eval和arguments两个标识符。
4、不能用delete删除显式声明的标识符，名称或具名函数。

使用注意

“use strict” 被放在一个javascript文件或函数的起始位置时才可以开启严格模式。
通常实践推荐在所有函数（或者最外层函数）的开始处加入“use strict” 指令启动严格模式，而绝不推荐在文件的开头处写全局“use strict” ，这时一些es校验器会给出警告。

含义

流行的浏览器有一些以自己的方式去理解css规范，这样会导致有的浏览器对css的解释与设计师的css定义初衷冲突，使得网页的样子在某些浏览器下能正确显示，但有些浏览器缺并没有按照设计师想要的样子显示出来，这就导致浏览器的兼容性问题，更糟的是，有些浏览器完全无视css的一些声明和属性。

学习笔记之gulp

发表于 2016-11-17 |

1、gulp.src(globs,[options])

src方法指定需要处理的源文件路径，gulp借鉴unix操作系统的管道(pipe)思想，前一级的输出变成后一级的输入。
“src/a/js”: 指定具体文件
““: 匹配所有文件 例：src/.js(包含src下所有的js文件)；
““ 匹配0个或多个子文件夹 例：src//js(包含src的0个或多个子文件夹下的js文件)；
“{}”:匹配多个属性 例：src/{a,b}.js(包含a.js和b.js文件)
src/.{jpg,png,gif}(src下所有的jpg,png,gif文件)；
“!” 排除文件 例：!/src/a.js(不包含src下的a.js文件)；

options:类型(可选):object,有3个属性buffer，read,base;

options.buffer:类型:boolean 默认：true 设置为false，将返回file.content的流并且不缓冲文件，处理大文件时非常有用；
options.read 类型boolean 默认：true 设置为false,将不执行读取文件操作，返回null
options.base: 类型boolean 默认：true 设置输出路径以某个路径的某个组成部分为基础向后拼接

2、gulp.dest(path[,options])

说明：dest方法是指定处理完文件后的输出路径；

path:类型(必填)：string or function 指定文件输出路径，或者定义函数返回文件输出路径；
options.cwd： 类型：String 默认：process.cwd()：前脚本的工作目录的路径 当文件输出路径为相对路径将会用到；
options.mode： 类型：String 默认：0777 指定被创建文件夹的权限；

gulp.task(name[,deps],fn)

说明：task定义一个gulp任务
name：类型： string 指定任务的名称
deps：类型： stringarray，该任务依赖的任务

fn:类型：function 该任务调用的插件操作

gulp.watch(glob[,opts],tasks)or gulp.watch(glob[,opts,cb])

说明：watch用于监听文件的变化，一修改就会执行指定的任务
glob：需要处理的源文件匹配符路径。类型：string or srtingarray
opts：类型（可选）object
tasks：类型（必填）stringarray 需要执行的任务的名称数组
cb（event）类型（可选）function 每个文件变化执行的回调函数

gulp监听less生成css

学习笔记之http

发表于 2016-11-17 |

无状态性

http协议是无状态的，同一个客户端第二次访问同一个服务器上的页面时，服务器无法知道这个客户端曾经访问过。服务器无法分辨不同的客户端。http无状态特性简化了服务器的设计，使服务器更容易支持大量并发的http请求。

过程

1、浏览器输入网址。
2、浏览器根据域名解析IP。
3、浏览器与web服务器建立一个tcp连接。
4、浏览器给web服务器发送一个http请求。
get请求 ／index.jsp?id=100&op=bind。
服务器返回301永久重定向响应。

状态码

1xx：信息性状态码，表示服务器已接收请求，客户端可以继续发送请求。

100: continue

2xx：成功状态码，表示服务器已成功接收到请求并处理。

200:ok表示客户端请求成功
204:no content 成功，但是不反悔任何实体的主体部分。

3xx：重定向状态码，表示服务器要求客户端重定向。

301:永久重定向，响应报文的location首部应有该资源的新URL。
303:请求的资源存在着另一个URL

4xx：客户端错误状态码，表示客户端请求有非法内容。

400 bad request 表示客户端请求有语法错误，不能被服务器所理解
401 表示未经过授权
403 表示服务器收到请求，但是拒绝提供服务，通常会在响应正文中给出
404 not found 请求的资源不存在

5xx 服务器错误状态码，表示服务器未能正常处理客户端的请求而出现的意外错误。

500: 表示服务器发生不可预期的错误，导致无法完成客户端的请求。
503 表示服务器当前不能够处理客户端的请求，在一段时间后可能恢复正常。
5、服务器发回一个HTML响应
6、释放TCP连接
7、浏览器解析HTML内容

https

https 端口443 http 端口 80
http运行在TCP之上，所有传输的内容都是明文，客户端和服务器端都无法验证对方的身份。
https是http运行在SSL／TLS之上，所有传输的内容经过对称加密，对称加密的密钥用服务器方的证书进行了非对称加密，客户端可以验证服务端的身份，如果配置了客户端验证，服务器也可以验证客户端身份。
